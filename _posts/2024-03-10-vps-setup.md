## Setting up a VPS with Portainer and Caddy to serve my web applications
Recently, I started considering serving my ML project demos online using custom urls, so that anyone can visit and use them. 
In the current golden age of ML applications, engineers are really spoiled for choice regarding their tools for creating a ML demo from python based frameworks like [Gradio](), [Streamlit] to traditional frontend frameworks like [React](), [NextJS]() etc.
Once you have got the application running on your local machine though, the question becomes how to host it online ?

Depending on your use case and compute requirements, the options range from completely serverless solutions by hyperscalers, to rentable Virtual Machines called Virtual Private Servers.
In this (opinionated) blog post I will discuss how I setup my VPS for serving some hobby projects.

### Virtual Private Server (VPS)
A virtual private server (VPS) is a virtual machine that you can rent from a hosting provider. Depending on the price, you can customize the storage, memory, CPU, GPU and bandwidth.
[Amazon EC2s]() are some of the most popular virtual machines to rent online, however they can become quite expensive. Personally I choose to host my services using [Hetzner Cloud]() because of their cheaper shared vCPU tier which provides good enough compute for hobby projects.

Once you have obtained a VPS, you will normally get an option to choose the base image that the server will use. This normally involves a choice between different operating systems that can (optionally) come pre-installed with certain applications.
I normally choose an Ubuntu OS, with docker pre-installed. You also would need a public IP that will be assigned to your VPS. Depending on your hosting service, it can come pckaged with the VPS, or you may need to purchase it separately.

### Setting up a Firewall

Before starting to run services, It's a good idea to make sure we have a basic firewall in place. We can use Uncomplicated Firewall (ufw) in ubuntu for this purpose. It is disabled by default, but before enabling it we need to do some additional steps.
First, let's list all the applications that we can regsiter with the firewall.
```bash
sudo ufw app list
```
Ideally this should just list OpenSSH in a fresh VPS. We need to add this application to the list of applications which are allowed by the firewall, so that we can stil use `ssh` to login after the firewall is active.
```bash
sudo ufw allow OpenSSH
```
Now finally we can enable the firewall
```bash
sudo ufw enable
```
Now w can check the status of the firewall using
```bash
ufw status
```
which should show that all connections except OpenSSH are blocked.

### Setting up Caddy
Once you have the VPS setup, the first thing I setup is a reverse proxy. A reverse proxy is a server that sits between clients (such as web browsers) and backend servers. It receives requests from clients on behalf of those servers and then forwards those requests to the appropriate backend server. We can use this reverse proxy to intercept requests sent to our server, and route them to appropriate services. This allows us to run multiple services and do load balancing, manage security as well as SSL encryption.

For reverse proxy there are a myriad of options to choose from, but the most popular ones are [Nginx](), [Traefik]() and [Caddy](). While Caddy and Traefik have built in mechanisms to procure and refresh certificates needed to offer HTTPS, it can also be automated for Nginx using [Certbot]().

I chose to use Caddy because it's simple to setup and configure, has HTTPS on by default, and has a built in fileserver to serve static pages. The easiest way to setup caddy is through a docker image.

To launch Caddy we need to do the following steps:

#### Step 1: Allow HTTP and HTTPS Connection through the Firewall
We had blocked all connections except ssh ealier. Now let's add a rule to allow HTTP and HTTPS connections as well as UDP connections through some standard ports, such that they can be intercepted by our reverse proxy.

```bash
sudo ufw allow proto tcp from any to any port 80,443
sudo ufw allow 443/udp
```

#### Step 2: Create a docker bridge network
We need to create a docker network using:
```bash
docker network create -d bridge caddy-vps
```

This creates a docker bridge network called `caddy-vps`. Now I can start both Caddy and other services that need reverse proxy in this network, so that they can communicate to each other while being isolated from the host system.

#### Step 3: Point your (sub)domains to the VPS
You need to point any domains/subdomains that you want to use for services running in the VPS towards the public IP attached to the VPS.

I used mostly subdomains, so needed to create multiple `A record` instances to point specific subdomains towards the public IP of my VPS.

#### Step 4: Create a Caddyfile
Caddyfile contains all configuration regarding the reverse proxy (which domain to route to which service) as well as info about the fileserver (which folder to serve as root for the static fileserver).

For me, the two routes I want to configure are a subdomain to access portainer instance as well as a static fileserver.
So my Caddyfile looks like this:

```
portainer.example.com {
    reverse_proxy portainer:9000
}

fileserver.example.com {
    root */srv
    file_server
}
```
where instead of `example.com`, I use my own domain.

#### Step 5: Create Caddy Docker Compose file

We create a caddy docker compose yaml file following the [official instructions]().

```yaml
version: '3.8'
services:
  caddy:
    image: caddy:2.7.6-alpine
    restart: unless-stopped
    ports:
      - 80:80
      - 443:443
      - 443:443/udp
    volumes:
      - $PWD/Caddyfile:/etc/caddy/Caddyfile
      - $PWD/static:/srv
      - caddy_data:/data
      - caddy_config:/config
    networks:
      - caddy-vps

networks:
  caddy-vps:
    external: true

volumes:
  caddy_data:
    driver: local
  caddy_config:
    driver: local
```
This docker file mounts a local directory as a volume which will be used as root for the static file serving.
It also uses docker volumes for data and config volumes required by the Caddy image. The host computers ports that we can receive HTTP and UDP connections from are directly mounted to the container. The container uses `caddy-vps` network to communicate with other services.

#### Step 4: Lanuch caddy !
To run caddy using the previous docker compose file we can use the command:
```bash
docker compose up --build -d
```
Which launches the services in detached mode.

### Portainer

Now that we have our Caddy up and running, we want a way to manage Caddy and other container we may choose to deploy in this VPS. [Portainer]() is an open-source tool that provides a user friendly UI to deploy, monitor and manage docker containers, images, volumes and networks.

Similar to caddy, the simplest way to launch a portainer instance is through docker. I useed the following docker compose yaml file:

```yaml
version: "3.8"
services:
  portainer:
    image: portainer/portainer-ce:latest
    volumes:
      - data:/data
      - /var/run/docker.sock:/var/run/docker.sock
    restart: unless-stopped
    networks:
      - caddy-vps

networks:
  caddy-vps:
    external: true

volumes:
  data:
    driver: local
```
Which launches a portainer instance in the same network as our Caddy instance. This allows Caddy to be able to function as a reverse proxy for this service. Now we can visit `portainer.example.com` subdomain and access the web gui of portainer to easily manage and monitor our containers.

### Conclusion
In this blog post we learned how to setup a VPS to deploy web applications online using Caddy and Portainer.
